# 비전공자용 설명서

날짜: `2026-04-24`
상태: 비전공자용 보조 문서
기준 문서: `AGENTS.md`
변경 기록: `patch.md`

이 문서는 개발자가 아닌 사람이 봐도 이 저장소가 무엇이고, 왜 정리했는지 이해할 수 있도록 설명하는 문서입니다.

파일명은 공개 저장소에서 다루기 쉽도록 영어로 유지하지만, 본문은 포트폴리오 제출자가 빠르게 읽을 수 있도록 한국어로 작성합니다.

## 이 저장소가 보여주는 것

이 저장소는 제조 운영 KPI 대시보드 데모입니다.

목적은 실제 회사 자료를 공개하는 것이 아니라, 대시보드 구조와 개발 방식, 데이터 흐름, 화면 구성, 테스트 정리 방식을 보여주는 것입니다.

즉, 포트폴리오에서 보여주고 싶은 핵심은 다음입니다.

- KPI 화면을 어떻게 구성했는지
- 생산, 유틸리티, 점검, 리포트 기능을 어떻게 나누었는지
- 로컬 화면과 서버 런타임을 어떻게 연결할 수 있게 만들었는지
- 실제 업무 데이터를 공개하지 않고도 데모가 돌아가도록 어떻게 정리했는지

## 공개 저장소에 넣으면 안 되는 것

포트폴리오용 공개 저장소에는 아래 자료가 들어가면 안 됩니다.

- 실제 회사명
- 실제 고객사명
- 직원 이름
- 실제 공장, 지점, 현장명
- 실제 설비명
- 실제 생산량, 전력, 가스, 폐수, 점검, 정산 값
- 원본 로고, 사진, 증빙 파일, 내부 캡처
- 특정 회사 지원 전략이나 채용 사이트 링크

대신 아래처럼 더미 이름을 사용합니다.

- `Target Company`
- `Previous Employer`
- `Client A`
- `Plant A`
- `Plant B`
- `Line Alpha`
- `Operator A`
- `Sample Equipment`
- `Synthetic KPI value`

## 2026-04-25 - 공개용 팀명과 공정명으로 정리

### 이번에 건드린 위치

```text
KPI-Demo/
|-- README.md                 공개 데모 설명
|-- AGENTS.md                 앞으로 지켜야 할 공개 이름 규칙
|-- kpi-runtime/demo/         화면에 들어가는 데모 데이터
|-- team-report/runtime/work/ 작업 이력과 팀 화면
`-- utility/                  생산, 검침, 리포트 관련 기능
```

### 무엇을 바꿨나

화면과 테스트에 남아 있던 실제 업무처럼 보이는 팀명, 파트명, 공정명, 설비명, 제품 코드를 공개용 가짜 이름으로 바꿨습니다.

예를 들면 팀은 `Line Alpha`, `Line Beta`처럼 보이게 했고, 공정은 `Process Alpha`, 설비는 `Demo Boiler A`, 제품은 `Product A`처럼 보이게 했습니다.

### 왜 중요한가

포트폴리오에서는 기능 구조와 구현 방식을 보여주면 됩니다. 실제 회사에서 쓰던 팀명이나 공정명이 그대로 보이면, 데이터 값이 가짜여도 내부 업무 구조를 공개하는 것처럼 보일 수 있습니다.

그래서 데이터 값뿐 아니라 이름 체계도 전부 공개 데모용으로 맞췄습니다.

### 확인한 내용

예전 회사명, 팀명, 공정명, 설비명, 제품 코드가 남아 있는지 다시 검색했습니다.

주요 자바스크립트 파일 문법 검사와 관련 테스트 `47`개도 통과했습니다.

마지막으로 전체 포맷 검사, 린트, 전체 테스트 `425`개까지 다시 돌려 통과를 확인했습니다.

정적 데모 화면도 브라우저로 직접 열어 봤고, 콘솔 오류나 예전 팀명·공정명·설비명 노출은 없었습니다.

공개 GitHub 리포에 올릴 때 `node_modules` 같은 설치 폴더와 개인 로컬 AI 설정 폴더가 같이 올라가지 않도록 `.gitignore`도 보강했습니다.

회사명, 옛 로고 경로, 예전 행동강령 표현도 다시 검색했습니다. 공개하면 위험한 회사 자료 패턴은 더 이상 나오지 않았습니다.

## 2026-04-25 - 서버용 문서 규칙을 공개 데모 기준으로 반영

### 이번에 건드린 위치

```text
KPI-Demo/
|-- AGENTS.md                  AI가 따라야 할 공식 규칙
|-- patch.md                   개발자가 보는 기술 변경 기록
`-- plain-english-companion.md 비전공자도 읽을 수 있는 쉬운 설명
```

### 무엇을 바꿨나

서버용으로 쓰던 문서 규칙 중 지금 공개 데모에도 필요한 부분만 가져와서 정리했습니다.

그냥 파일을 복사하지는 않았습니다. 서버용 문서에는 운영 서버 기준 표현이나 예전 폴더명이 섞일 수 있기 때문에, 현재 공개 포트폴리오 저장소인 `KPI-Demo`에 맞게 다시 썼습니다.

이번에 반영한 핵심은 다음입니다.

- 공식 규칙은 `AGENTS.md` 하나만 기준으로 봅니다.
- `patch.md`는 기술 변경 기록일 뿐, 공식 규칙 문서는 아닙니다.
- `plain-english-companion.md`는 비전공자도 이해할 수 있게 쉽게 풀어쓴 보조 문서입니다.
- 다른 서버용 클론이나 백업 폴더에서 내용을 가져올 때는 그대로 복사하지 않고, 현재 폴더 구조와 공개 기준에 맞게 바꾼 뒤 반영합니다.
- 실제 회사명, 현장명, 직원명, 내부 데이터, 임시 실행 파일은 공개 데모에 넣지 않습니다.
- 앞으로 변경 기록에는 어느 폴더와 기능 영역을 건드렸는지 먼저 적습니다.

### 왜 중요한가

포트폴리오 공개 저장소에서는 “기능이 있다”만큼이나 “자료를 안전하게 정리했다”가 중요합니다.

서버용 문서 내용을 그대로 가져오면 예전 이름, 내부 운영 기준, 실제 업무 흔적이 다시 섞일 수 있습니다. 그래서 필요한 규칙만 현재 공개 데모 구조에 맞게 흡수했습니다.

### 확인한 내용

문서 3개만 수정했고, 실제 업무 데이터나 회사 자료는 추가하지 않았습니다.

민감 키워드 검색과 문서 읽기 확인을 완료했고, 결과는 기술 변경 기록인 `patch.md`에 함께 남겼습니다.

## 2026-04-25 - 데모 실행 명령과 예시 로그인 정리

### 이번에 건드린 위치

```text
KPI-Demo/
|-- README.md                 처음 보는 사람이 보는 실행 안내
|-- commands/                 더블클릭해서 데모를 열고 닫는 명령
`-- kpi-runtime/internal-server/
    |-- src/routes/auth-routes.js              로그인 화면
    `-- scripts/windows/                       선택형 서버 런타임 실행 스크립트
```

### 무엇을 바꿨나

처음 보는 사람이 포트와 로그인 정보를 헷갈리지 않도록 정리했습니다.

공개용 정적 데모는 아래 명령으로 열 수 있습니다.

- `commands\open-kpi-demo.cmd`
- `commands\close-kpi-demo.cmd`

포트도 문서와 명령 출력에 맞춰 적었습니다.

- 정적 데모 화면: `5500`
- 선택형 Node 서버 런타임: `3100`
- 선택형 PostgreSQL DB: `5434`
- 명령 콘솔: `3215`, 사용 중이면 `3216`부터 `3218`

서버 런타임 로그인 화면에는 예시 계정을 바로 알 수 있게 표시했습니다.

- 아이디: `1234`
- 비밀번호: `1234`

입력칸에도 기본으로 `1234`가 들어가게 해서, 처음 보는 사람이 무엇을 입력해야 하는지 바로 알 수 있습니다.

### 왜 중요한가

포트폴리오를 보는 사람은 내부 운영 구조를 모릅니다.

그래서 “어느 주소로 들어가야 하는지”, “로그인 화면에서 뭘 입력해야 하는지”를 화면과 문서에 바로 보여주는 편이 좋습니다. 이번 변경은 실제 보안 계정이 아니라 공개 데모용 예시 계정 기준입니다.

### 확인한 내용

로그인 화면에 `1234 / 1234` 예시가 들어가는지 테스트를 추가했고 통과했습니다.

명령 스크립트와 문서의 포트 안내도 같은 기준으로 맞췄습니다. 전체 테스트는 `425`개가 통과했습니다.

## 2026-04-24 - 공개 문서 정리

### 무엇을 바꿨나

포트폴리오로 공개해도 되도록 문서를 정리했습니다.

이전 문서에는 깨진 한글 인코딩, 특정 회사 지원용 리서치, 실제 업무 맥락에 가까운 표현이 섞여 있었습니다. 이런 내용은 삭제하거나 일반적인 표현으로 바꿨습니다.

### 바뀐 파일

- `AGENTS.md`: 저장소 작업 규칙과 공개 기준을 정리했습니다.
- `patch.md`: 기술 변경 기록을 정리했습니다.
- `plain-english-companion.md`: 비전공자도 이해할 수 있는 설명서로 정리했습니다.
- `agent.md`: 특정 회사만 겨냥한 내용이 남지 않게 정리했습니다.
- `deep-research-report.md`: 회사 리서치 자료가 남지 않게 정리했습니다.

### 왜 중요한가

포트폴리오는 실제 회사 정보를 공개하는 곳이 아닙니다.

보여줘야 하는 것은 “어떤 문제를 어떻게 구조화해서 해결했는지”입니다. 그래서 회사명, 현장명, 사람 이름, 실제 수치 대신 더미 데이터와 일반적인 이름을 사용해야 합니다.

### 확인한 내용

마크다운 문서에서 회사명, 현장명, 채용 사이트, 사람 이름 패턴이 남아 있는지 검색했습니다.

검색 결과, 공개하면 안 되는 민감 키워드는 남아 있지 않았습니다.

## 2026-04-24 - 코드와 테스트 라벨 익명화

### 무엇을 바꿨나

문서뿐 아니라 실제 코드, 테스트, 스크립트에 남아 있던 현장명과 사람 이름도 정리했습니다.

기존 실제 업무 라벨은 `Plant A`, `Plant B`, `Operator A` 같은 공개용 더미 라벨로 바꿨습니다.

또 서버 쪽 이름도 내부 회사 시스템처럼 보이지 않도록 `KPI Demo Runtime` 기준으로 바꿨습니다.

### 바뀐 범위

- 계량, 리포트, 팀 업무, 런타임 관련 소스 코드
- 관련 스모크 테스트
- 런타임 패키지 메타데이터
- CI 워크플로 이름
- Plant B 팀 개요 카드 파일명과 테스트 파일명

### 왜 중요한가

코드 안에 실제 회사나 현장 이름이 남아 있으면, 문서만 정리해도 공개 저장소로는 부족합니다.

면접이나 포트폴리오 제출에서는 “실제 데이터를 숨겼는지”도 중요하게 봅니다. 그래서 화면에 보이는 이름뿐 아니라, 테스트와 파일명까지 더미 이름으로 맞췄습니다.

### 확인한 내용

코드, 테스트, 스크립트, 파일명에서 민감 키워드를 다시 검색했습니다.

검색 결과, 제거 대상 회사명, 현장명, 사람 이름 패턴은 남아 있지 않았습니다.

추가로 포맷 검사, 린트, 주요 스모크 테스트, 전체 테스트를 실행했습니다.

전체 테스트는 `424`개가 통과했습니다.
